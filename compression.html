In deep learning for compression tasks, the dual representation of data as both floating-point and integer (long) types serves multiple critical purposes. Let's dive deeper into why this approach is essential for our lossless compression model.

The float version, reshaped to (1200, 1, 8, 16), is fundamental to the neural network's learning process. Neural networks operate on continuous values, and floating-point representations allow for smooth gradient calculations during backpropagation. When our convolutional layers process the spatial patterns in the driving video tokens, they need this continuous representation to detect subtle features and patterns. The shape itself is significant: 1200 frames maintain the temporal sequence, the single channel represents our token values, and the 8x16 spatial dimensions preserve the structural relationships within each frame.

The long (integer) version, reshaped to (1200, -1), serves a different but equally crucial role. In essence, we're treating each token as a categorical value - a discrete class index from 0 to 1023 (since we're using 10-bit tokens). This representation is essential for our loss function's classification aspects and, more importantly, for verifying lossless compression. When we check if our compression is truly lossless, we need exact integer matches between our predicted tokens and the original ones. The flattened shape (-1) simplifies this comparison process and works well with PyTorch's classification loss functions like cross-entropy, which expect class indices rather than continuous values.

The interplay between these two representations is what makes our compression system work. During training, the float version helps the model learn smooth, continuous representations of the token patterns, while the long version ensures we can verify exact reconstruction. This duality is particularly important in our case because we're not just aiming for good compression - we're demanding perfect reconstruction while achieving a specific compression ratio (3.6x). The float version helps us optimize for compression through gradient descent, while the long version acts as our "quality control," ensuring we maintain the lossless requirement of our system.

Here's how our loss function uses both versions:

class HybridLoss(nn.Module):
    def __init__(self, compression_target=3.6, lossless_weight=100.0):
        super().__init__()
        self.compression_target = compression_target
        self.lossless_weight = lossless_weight
    
    def forward(self, predictions, confidence, residuals, true_tokens):
        batch_size, seq_len = predictions.shape[:2]
        
        # Reshape predictions (from float computations) and targets (long indices)
        predictions = predictions.view(batch_size * seq_len, -1)  # From float processing
        true_tokens = true_tokens[:, :, 0].reshape(-1)           # Long indices for matching
        
        # Token prediction accuracy using long indices
        pred_tokens = predictions.argmax(dim=1)                  # Convert to token indices
        token_match_loss = (pred_tokens != true_tokens).float().mean() * self.lossless_weight
        
        # Distribution matching using float predictions
        pred_loss = F.nll_loss(predictions, true_tokens)        # Uses float logits with long targets
        
        # Compression rate calculation using float values
        original_bits = true_tokens.numel() * 10
        pred_entropy = -(predictions.exp() * predictions).sum(dim=1).mean()
        compressed_bits = true_tokens.numel() * pred_entropy
        compression_rate = original_bits / compressed_bits
        
        # Final loss combining both versions
        rate_loss = F.mse_loss(compression_rate, 
                              torch.tensor(self.compression_target, device=predictions.device))
        total_loss = token_match_loss + pred_loss + rate_loss
        
        return total_loss, compression_rate

Key points about how the versions interact:

  • Float predictions generate probabilities for each token
  • Long targets provide ground truth for exact matching
  • We combine both for a loss that ensures both accuracy and compression

Let's dive into LogSoftmax and NLL (Negative Log-Likelihood) loss, which form a crucial partnership in our model:

LogSoftmax combines two operations: it first applies the softmax function to convert our raw model outputs into probabilities (they sum to 1), then takes the natural logarithm of these probabilities. We use it in our token_predictor:

self.token_predictor = nn.Sequential(
    nn.Linear(2048, 1024),    # Raw scores for each possible token
    nn.LogSoftmax(dim=-1)     # Convert to log probabilities
)

This pairs perfectly with NLL loss in our loss function:

pred_loss = F.nll_loss(predictions, true_tokens)

Why use this combination? Three key reasons:

  1. Numerical stability - computing log(softmax(x)) directly can cause underflow/overflow, but LogSoftmax uses a more stable implementation
  2. Computational efficiency - combining softmax and log in one operation is faster
  3. Perfect match with NLL loss - since we're already working with log probabilities, NLL just needs to select and negate the predicted probability of the true token

Loss calculation process?

The probability flow through our model represents a carefully orchestrated sequence of transformations that enable both effective learning and precise token prediction. Starting with the raw logits from our Linear layer, we get unconstrained scores for each possible token - these scores can be any real number and don't have any direct probabilistic interpretation. In our case, with a vocabulary of 1024 tokens, we're producing 1024 such scores for each position in our sequence.

The LogSoftmax transformation then performs two critical operations in a numerically stable way. First, it applies the softmax function, which exponentiates each score and normalizes them to create a valid probability distribution (summing to 1). Then, it immediately takes the logarithm of these probabilities. This log space representation is crucial because it prevents numerical underflow when dealing with very small probabilities, which are common when working with a large vocabulary of 1024 tokens. For example, a raw logit of 3.2 might become a log probability of 0.0, representing the highest probability in our distribution.

When these log probabilities meet our NLL loss calculation, we see the elegant simplicity of this approach. The NLL loss simply needs to look up the log probability assigned to the correct token and negate it. This negative log probability becomes our basic loss value - the lower the better, meaning the model assigned a higher probability to the correct token. For instance, if our model assigned a probability of 0.865 (log probability of 0.0) to the correct token, this represents very confident and accurate prediction, resulting in a low loss value.

This probability flow ultimately feeds into our hybrid loss function, where it combines with our exact matching and compression rate objectives. The log probabilities play a dual role here: they contribute to the basic prediction accuracy through the NLL loss, but they also factor into our entropy calculations for determining the compression rate. This dual use of the probability distribution is what allows our model to simultaneously optimize for both accurate prediction and efficient compression.

Model Architecture

Let's dive deeper into our model architecture, which combines spatial and temporal understanding to achieve lossless compression:

class OptimizedContextModel(nn.Module):
    def __init__(self, chunk_size=15):
        super().__init__()
        
        # Spatial Encoder: Learns local patterns in each frame
        self.spatial_encoder = nn.Sequential(
            nn.Conv2d(1, 32, kernel_size=3, padding='same'),  # Initial feature extraction
            nn.ReLU(),                                        # Non-linearity for pattern detection
            nn.Conv2d(32, 64, kernel_size=3, padding='same')  # Higher-level features
        )
        
        # Temporal Understanding: Captures sequence patterns
        self.temporal_lstm = nn.LSTM(
            input_size=64 * 8 * 16,  # Flattened spatial features (64 channels * 8x16 frame)
            hidden_size=1024,        # Rich temporal context
            num_layers=1,            # Single layer for memory efficiency
            bidirectional=True,      # Look both forward and backward in time
            batch_first=True         # Batch dimension first for easier handling
        )
        
        # Prediction Heads
        self.token_predictor = nn.Sequential(
            nn.Linear(2048, 1024),   # 2048 = bidirectional LSTM (2*1024)
            nn.LogSoftmax(dim=-1)    # Convert to log probabilities
        )
        
        self.confidence_estimator = nn.Sequential(
            nn.Linear(2048, 1),
            nn.Sigmoid()             # Output confidence between 0 and 1
        )

The architecture works in three stages:

  1. Spatial Understanding: The convolutional layers learn local patterns within each frame. The first conv layer detects basic features (edges, gradients), while the second finds more complex patterns. The 'same' padding ensures we maintain spatial dimensions.

  2. Temporal Context: The bidirectional LSTM processes these spatial features over time, learning both forward and backward dependencies. This is crucial for compression as it helps predict tokens based on surrounding context.

  3. Dual Output: We predict both tokens (with probabilities) and confidence scores. This dual output helps balance between compression ratio and lossless reconstruction requirements.

def train_model(train_loader, val_loader, model, criterion, optimizer, scheduler, metrics):
    global_step = 0
    best_loss = float('inf')
    
    for epoch in range(CONFIG['num_epochs']):
        print(f"\nEpoch {epoch+1}/{CONFIG['num_epochs']}")
        model.train()
        
        for batch_idx, (data, target) in enumerate(train_loader):
            # Memory management
            if batch_idx % 10 == 0:
                torch.cuda.empty_cache()
            
            # Forward pass with both data types
            data, target = data.to(device), target.to(device)
            predictions, confidence = model(data)
            loss, rate = criterion(predictions, confidence, None, target)
            
            # Optimization step
            optimizer.zero_grad()
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
            optimizer.step()
            
            # Progress tracking
            metrics.update(
                train_loss=loss.item(),
                compression_rate=rate.item(),
                is_lossless=torch.all(predictions.argmax(-1) == target[:,:,0]),
                iteration=global_step
            )

Challenges and Solutions

In developing our lossless compression model, we faced three major challenges that required careful consideration and innovative solutions.

Memory management proved to be our first significant hurdle. The task of processing 1200-frame sequences initially overwhelmed our GPU memory, even with modern hardware. We addressed this through a chunk-based processing strategy, breaking sequences into manageable 15-frame segments. This approach, combined with regular memory cleanup through torch.cuda.empty_cache(), allowed us to maintain stable training without sacrificing model capacity. The key insight was that temporal patterns could still be effectively learned across chunk boundaries, while dramatically reducing peak memory usage.

Training stability presented our second major challenge, particularly in balancing the competing goals of high compression ratios and perfect reconstruction. Our solution centered on dynamic loss weighting, with a substantial weight (100.0) assigned to the lossless reconstruction term. This weighting scheme effectively created a hierarchy of objectives, where the model first learns to reconstruct perfectly, then optimizes for compression within that constraint. The addition of a confidence estimator further stabilized training by allowing the model to express uncertainty about its predictions.

The third challenge involved convergence issues, where the model would often stall at suboptimal solutions. We implemented a three-pronged approach: gradient clipping prevented explosive updates that could derail training, confidence estimation helped the model focus on high-probability predictions, and an adaptive learning rate schedule automatically adjusted optimization parameters when progress plateaued. This combination proved crucial in achieving consistent convergence to solutions that met both our compression target and lossless requirement.

Future Work and Improvements

Our initial results suggest several promising directions for extending and improving this lossless compression system. The most immediate opportunity lies in leveraging our unexpectedly efficient memory usage. With only 22% GPU utilization, we have substantial headroom to scale up the model's capacity. This could involve increasing the number of channels in our spatial encoder from 32/64 to potentially 64/128, or expanding the LSTM's hidden dimensions beyond the current 1024 units.

The training dynamics we observed suggest that architectural modifications could yield significant improvements. One compelling direction would be to implement a hierarchical chunking strategy, where the model processes sequences at multiple temporal scales simultaneously. This could help capture both fine-grained frame-to-frame patterns and longer-range dependencies that might improve compression ratios. Additionally, replacing the simple bidirectional LSTM with a more sophisticated transformer-based architecture might better capture complex temporal relationships.

The rapid training convergence (6 minutes for 10 epochs) opens up possibilities for more extensive experimentation. We could explore curriculum learning approaches, starting with shorter sequences and gradually increasing length, or implement more sophisticated scheduling strategies beyond our current ReduceLROnPlateau approach. The quick iteration time also suggests we could afford to implement cross-validation or more robust evaluation protocols.

Perhaps most intriguingly, we could explore adaptive chunk sizing based on sequence content. Instead of our fixed 15-frame chunks, the model could learn to dynamically adjust chunk sizes based on the complexity and predictability of different sequence segments. This could potentially help us bridge the gap between our current 3.2x compression ratio and the target of 3.6x while maintaining our critical lossless requirement.
